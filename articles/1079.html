1) Why-read

LogRouter proposes a two-level cost-aware router for log QA that achieves 88.4% mean accuracy and 55% latency reduction over a fixed-32B baseline by dispatching queries to LLM-bypass paths or appropriately-sized generators.

2) CRGP

C — Context

Production log analytics in self-hosted environments requires natural-language access to massive log streams, but routing every query through a large language model is costly and often suboptimal. Existing log-RAG prototypes operate on static logs or rely on cloud APIs, and prior routing work (e.g., FrugalGPT, Adaptive-RAG) assumes every query ultimately reaches an LLM, missing the opportunity for LLM-bypass paths.

R — Related work

  • LLMLogAnalyzer [1] proposes a clustering-based log chatbot with a router, but every query traverses a large generative model.
  • FrugalGPT [2] cascades LLM APIs with early-exit confidence thresholds, achieving cost reduction but still routing all queries through an LLM.
  • Adaptive-RAG [8] trains a classifier to route among retrieval strategies, but does not consider LLM-bypass paths.
  • LogGPT [12] and Mudgal & Wouhaybi [11] apply ChatGPT to log analysis but rely on commercial cloud APIs and have scalability limitations.

G — Gap

Prior log QA systems either route every query through a single LLM (wasting resources on simple queries) or rely on external cloud APIs (raising privacy/cost concerns). No existing work provides a production-grade, on-premises log QA system with explicit LLM-bypass paths and dynamic model-size selection for the semantic path.

P — Proposal

LogRouter is an end-to-end log QA system with a two-level cost-aware router: Level-1 dispatches queries to one of four paths (direct response, Druid keyword search, template lookup with SQL generation, pgvector semantic retrieval) using a regex-based keyword vocabulary, and Level-2 selects a 14B or 32B generator for the semantic path. The system is deployed on TUBITAK BILGEM's national big data platform with a PySpark-based Drain3 ingestion pipeline and dual-index storage in Apache Druid and PostgreSQL with pgvector.

3) Figures

Figure extraction is performed separately and inlined post-generation.

4) Experiments

Main results

On four LogHub datasets (70 questions total), the router achieves 88.4% mean accuracy (94.7% on Linux). The full pipeline attains mean ROUGE-1 of 0.373, BERTScore of 0.879, RAGAS Faithfulness of 0.779, and end-to-end latency of 18.6 s. In offline comparison, the routed system reduces mean latency by 55% vs. Fixed-32B (46.3 s vs. 102.1 s) while preserving Answer Correctness within 5.8 points and exceeding Fixed-14B on RAGAS Faithfulness across every dataset.

Ablation

Removing the router (always-32B) increases latency by 55% with marginal quality gain; removing Drain template extraction reduces routing precision significantly but does not affect generation quality, showing Drain contributes almost entirely to routing precision.

Limitations

['Evaluation is limited to 70 questions across four datasets, which may not capture the full diversity of real-world log queries.', 'The regex-based L1 router relies on handcrafted patterns (P0-P6) that may require maintenance as query patterns evolve.', 'The system is deployed on a specific platform (TUBITAK BILGEM), and generalizability to other big data environments is not demonstrated.']

5) Why it matters

  1. Demonstrates that cost-aware routing with LLM-bypass paths can recover most of the quality of an always-32B configuration at less than half the latency, a practical insight for resource-constrained deployments.
  2. Shows that Drain template extraction primarily benefits routing precision rather than generation quality, suggesting that lightweight parsing may suffice for retrieval in log QA.
  3. Provides a blueprint for on-premises log QA systems that avoid external API calls, addressing privacy and cost constraints in public-sector or enterprise settings.

6) Next steps

  • [ ] Evaluate LogRouter on a larger and more diverse set of log queries (e.g., from production incidents) to validate generalizability.
  • [ ] Explore learned routing policies (e.g., lightweight classifier) to replace the regex-based L1 router for better adaptability.
  • [ ] Investigate the impact of different embedding models and chunking strategies on retrieval quality for log data.
  • [ ] Extend the system to support multi-turn conversational log analysis and incorporate user feedback for router adaptation.
