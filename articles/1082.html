1) Why-read

Key claim: predictive prefetching of retrieval based on semantic precursors can reduce RAG latency by 43.5% while maintaining answer quality; key observation: retrieval needs are preceded by identifiable semantic precursors 8-16 tokens before uncertainty becomes critical.

2) CRGP

C — Context

Retrieval-Augmented Generation (RAG) improves factual grounding in LLMs but suffers from substantial latency due to synchronous retrieval. Recent work explores asynchronous retrieval, but existing approaches rely on heuristic coordination and assume stable information demands during decoding, which breaks in complex, multi-domain settings.

R — Related work

  • FLARE (Jiang et al., 2023): reactively triggers retrieval based on uncertainty signals, blocking generation.
  • Self-RAG (Asai et al., 2024): dynamically triggers retrieval reactively, blocking generation.
  • DRAGIN (Su et al., 2024): reactively triggers retrieval based on attention-weighted signals, blocking generation.
  • TeleRAG (Lin et al., 2025): asynchronous retrieval with fixed lookahead window, triggering at predetermined intervals.
  • PipeRAG (Jiang et al., 2025): asynchronous retrieval using generated (potentially stale) context as queries.

G — Gap

Existing asynchronous retrieval methods rely on manually designed heuristics (e.g., fixed intervals, stale tokens) that assume stable information demands during generation. This assumption fails in complex multi-domain settings where topic boundaries blur and entity references evolve mid-generation, leading to irrelevant prefetched content and increased latency.

P — Proposal

The paper proposes an asynchronous retrieval framework with three components: a retrieval predictor (forecasts when retrieval is needed using entropy, attention, and discourse markers), a context monitor (determines optimal waiting horizon for query construction), and a query generator (constructs queries aligned with anticipated needs). These components are pretrained jointly on historical generation traces and adapted online via policy gradient with action-specific feedback.

4) Experiments

Main results

On HotpotQA, 2WikiMultiHopQA, Natural Questions, and TriviaQA: 43.5% end-to-end latency reduction, 62.4% improvement in time-to-first-token, 31% fewer retrievals per 1K tokens, while maintaining answer quality within 1% of synchronous RAG baselines.

Ablation

The retrieval predictor's learned timing (vs. fixed interval or reactive) is critical: predictive timing yields 43.5% latency reduction vs. 28% for fixed-interval asynchronous baselines, and avoids the 15% quality drop seen with stale-query methods like PipeRAG.

Limitations

['Requires access to LLM hidden states and attention weights at inference time, which may not be available for all model serving frameworks.', 'Assumes retrieval latency in 100-500 ms range; benefits may diminish for very fast or very slow retrieval.', 'Online adaptation via policy gradient may introduce training instability and requires careful reward design.']

5) Why it matters

  1. Directly addresses the latency bottleneck in RAG, which is critical for production deployment and user experience.
  2. Demonstrates that semantic precursors in generation dynamics can be exploited for predictive prefetching, a novel insight for asynchronous retrieval.
  3. Provides a principled framework (predictor, monitor, generator) that can be adapted to other latency-sensitive LLM serving scenarios.
  4. Shows that predictive prefetching can reduce retrieval count while maintaining quality, which has implications for cost and API rate limits.

6) Next steps

  • [ ] Evaluate the framework on more diverse multi-domain and long-form generation tasks to test robustness of semantic precursor detection.
  • [ ] Explore integration with other system-level optimizations (e.g., KV-cache, speculative decoding) to further reduce end-to-end latency.
  • [ ] Investigate the transferability of the pretrained components to different LLM architectures and sizes.
  • [ ] Design a lightweight variant that does not require full hidden state access, for broader compatibility.
