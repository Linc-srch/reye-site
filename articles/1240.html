Laravel-Lang PHP Packages Compromised to Deliver Cross-Platform Credential Stealer

Laravel-Lang PHP Packages Compromised to Deliver Cross-Platform Credential Stealer

supply-chain-attack
启示 · Insight

For a system security researcher focused on software supply-chain attacks, this incident demonstrates a sophisticated compromise of the release pipeline (organization-level credentials or automation infrastructure) rather than a single malicious commit. The use of Composer autoload.files for automatic execution on every PHP request is a notable persistence mechanism. The stealer's breadth—targeting cloud metadata endpoints, Kubernetes tokens, CI/CD secrets, and cryptocurrency wallets—shows attackers are systematically harvesting credentials that enable lateral movement and privilege escalation in cloud-native environments. The cross-platform payload (PHP dropper + VBScript on Windows, exec() on Linux/macOS) and the use of AES-256 encryption for exfiltration with self-deletion indicate a well-engineered, forensically-aware threat.

对于专注于软件供应链安全的系统安全研究员而言，此事件展示了对手对发布流水线（组织级凭据或自动化基础设施）的复杂攻陷，而非单个恶意提交。利用 Composer 的 autoload.files 在每个 PHP 请求上自动执行是一种值得注意的持久化机制。窃取器的广度——针对云元数据端点、Kubernetes 令牌、CI/CD 密钥和加密货币钱包——表明攻击者正在系统性地收集凭据，以便在云原生环境中实现横向移动和权限提升。跨平台载荷（PHP 投放器 + Windows 上的 VBScript，Linux/macOS 上的 exec()）以及使用 AES-256 加密进行外泄并自删除，表明这是一个精心设计、具有取证意识的威胁。

Summary

Multiple Laravel-Lang PHP packages were compromised via automated mass-tagging to deliver a cross-platform credential stealer that harvests cloud tokens, cryptocurrency wallets, browser credentials, and CI/CD secrets.

摘要

多个 Laravel-Lang PHP 包通过自动化批量打标签被攻陷，用于分发跨平台凭证窃取器，窃取云令牌、加密货币钱包、浏览器凭据和 CI/CD 密钥。

Deep Note

Key Points

  • Compromised packages: laravel-lang/lang, laravel-lang/http-statuses, laravel-lang/attributes, laravel-lang/actions.
  • Over 700 versions published in rapid succession on May 22-23, 2026, indicating automated mass-tagging.
  • Malicious code in src/helpers.php, registered in composer.json autoload.files for automatic execution on every PHP request.
  • Dropper contacts flipboxstudio[.]info to fetch a ~5,900 line PHP credential stealer with 15 collector modules.
  • Stealer targets: cloud metadata (AWS/GCP/Azure), Kubernetes tokens, CI/CD secrets (Jenkins, GitLab, GitHub Actions, etc.), cryptocurrency wallets, browser credentials, password managers, VPN configs, SSH keys, and more.
  • Uses AES-256 encryption for exfiltration and self-deletes to limit forensic evidence.
  • Cross-platform: VBScript launcher on Windows, exec() on Linux/macOS.

Context & Contrast

This attack is more sophisticated than typical PHP package typosquatting or dependency confusion. It compromises the legitimate release pipeline of a well-known organization (Laravel-Lang), using automated mass-tagging to inject backdoors into many versions simultaneously. The credential stealer's scope is unusually broad, covering cloud, CI/CD, crypto, and desktop applications, reflecting a shift toward targeting developer/infrastructure credentials rather than just end-user data. The use of Composer autoload.files for persistence is a novel vector in PHP supply-chain attacks.

Why It Matters

This incident highlights the critical risk of compromised release infrastructure in the PHP ecosystem, which can silently backdoor hundreds of package versions. The stealer's comprehensive credential harvesting directly threatens cloud-native and DevOps environments, enabling attackers to pivot from a single package dependency to full cloud account compromise. The cross-platform nature and forensic countermeasures (self-deletion, AES encryption) make detection and attribution harder. For system security researchers, this underscores the need for stronger supply-chain integrity measures (e.g., signed releases, CI/CD pipeline hardening, and runtime monitoring of autoloaded files).

Critical Take

The article relies on reports from Socket and Aikido Security; independent verification of the malicious payload and the full scope of compromised packages is needed. The claim of 'organization-level credentials' compromise is speculative—the exact vector (e.g., leaked token, compromised CI/CD, insider) is not confirmed. The stealer's ability to bypass Chromium's App-Bound Encryption (ABE) is mentioned but not detailed; the actual mechanism may be limited to specific versions or configurations. The number of affected users/sites is not disclosed, so the real-world impact remains unclear.

What to Watch

  • Follow-up disclosure from Laravel-Lang maintainers on the root cause and remediation steps.
  • Analysis of the exfiltration server (flipboxstudio[.]info) and any associated infrastructure.
  • Detection rules (YARA, Sigma) for the malicious helpers.php and the stealer payload.
  • Broader trend: similar attacks targeting other PHP package maintainers or using Composer autoload.files for persistence.
  • Impact assessment: number of sites that pulled the compromised versions and potential downstream breaches.
深度笔记

要点

  • 被攻陷的包：laravel-lang/lang、laravel-lang/http-statuses、laravel-lang/attributes、laravel-lang/actions。
  • 2026年5月22日至23日期间，快速连续发布了超过700个版本，表明是自动化批量打标签。
  • 恶意代码位于 src/helpers.php，通过 composer.json 的 autoload.files 注册，在每个 PHP 请求上自动执行。
  • 投放器联系 flipboxstudio[.]info 获取约5900行的 PHP 凭证窃取器，包含15个收集模块。
  • 窃取器目标：云元数据（AWS/GCP/Azure）、Kubernetes 令牌、CI/CD 密钥（Jenkins、GitLab、GitHub Actions 等）、加密货币钱包、浏览器凭据、密码管理器、VPN 配置、SSH 密钥等。
  • 使用 AES-256 加密进行外泄，并自删除以限制取证痕迹。
  • 跨平台：Windows 上使用 VBScript 启动器，Linux/macOS 上使用 exec()。

背景与对比

此攻击比典型的 PHP 包域名抢注或依赖混淆更为复杂。它攻陷了知名组织（Laravel-Lang）的合法发布流水线，利用自动化批量打标签同时向多个版本注入后门。凭证窃取器的范围异常广泛，涵盖云、CI/CD、加密货币和桌面应用，反映了攻击目标从终端用户数据向开发者/基础设施凭据的转变。利用 Composer 的 autoload.files 实现持久化是 PHP 供应链攻击中的一个新向量。

重要性

此事件凸显了 PHP 生态系统中发布基础设施被攻陷的关键风险，可能悄无声息地在数百个包版本中植入后门。窃取器的全面凭证收集直接威胁云原生和 DevOps 环境，使攻击者能够从单个包依赖转向完全云账户攻陷。跨平台特性和取证对抗措施（自删除、AES 加密）增加了检测和归因的难度。对于系统安全研究员而言，这强调了需要更强的供应链完整性措施（例如签名发布、CI/CD 流水线加固以及自动加载文件的运行时监控）。

批判性视角

文章依赖 Socket 和 Aikido Security 的报告；需要独立验证恶意载荷和被攻陷包的完整范围。关于‘组织级凭据’被攻陷的说法是推测性的——确切向量（例如泄露的令牌、被攻陷的 CI/CD、内部人员）尚未确认。窃取器绕过 Chromium 的 App-Bound Encryption (ABE) 的能力被提及但未详细说明；实际机制可能仅限于特定版本或配置。受影响用户/站点的数量未披露，因此实际影响仍不明确。

后续关注

  • Laravel-Lang 维护者关于根本原因和修复步骤的后续披露。
  • 对外泄服务器 (flipboxstudio[.]info) 及相关基础设施的分析。
  • 针对恶意 helpers.php 和窃取器载荷的检测规则（YARA、Sigma）。
  • 更广泛的趋势：针对其他 PHP 包维护者的类似攻击，或利用 Composer autoload.files 实现持久化。
  • 影响评估：拉取被攻陷版本的站点数量及潜在的下游入侵。
