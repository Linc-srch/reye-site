1) Why-read

AutoVecCoder proposes a framework combining automated data synthesis (VecPrompt) and reinforcement learning (VecRL) to teach LLMs to generate explicitly vectorized SIMD code, achieving SOTA on SimdBench and sometimes surpassing -O3 compiler optimizations.

2) CRGP

C — Context

Vectorization via SIMD is critical for high-performance computing, but compiler auto-vectorization often fails due to conservative static analysis, forcing developers to write explicit intrinsics manually. LLMs have excelled at general code generation but struggle with explicit vectorization due to scarce training data and strict hardware constraints.

R — Related work

  • VecTrans uses LLM agents to rewrite loops for better compiler auto-vectorization.
  • LLM-Vectorizer translates scalar C to vectorized code via zero-shot prompting, achieving up to 9.4x speedup on TSVC.
  • SimdBench provides a multi-architecture benchmark for evaluating vectorization correctness and performance.
  • IntrinTrans incorporates execution-guided optimization for cross-architecture intrinsic translation.
  • Afterburner applies GRPO with execution speed reward to optimize general code efficiency.
  • KernelBench and TritonBench evaluate operator-level GPU kernel efficiency, often using RL for DSL code generation.

G — Gap

Existing LLM-based vectorization approaches rely on zero-shot prompting or modular workflows without systematically training models to understand SIMD intrinsic semantics. They lack a dedicated training framework that injects domain-specific knowledge and aligns generation with execution efficiency, leading to suboptimal performance on explicit vectorization tasks.

P — Proposal

AutoVecCoder integrates VecPrompt, an automated data synthesis pipeline using RAG to inject SIMD intrinsic knowledge from real-world code and synthetic schemata, and VecRL, a GRPO-based reinforcement learning framework with a correctness-gated performance reward. The resulting 8B model achieves state-of-the-art results on SSE and AVX subsets of SimdBench, outperforming larger closed-source models and sometimes beating -O3 compiler optimizations.

4) Experiments

Main results

On SimdBench SSE subset, AutoVecCoder-8B achieves 78.5% pass@1 correctness and 1.12x average speedup over -O3 baseline; on AVX subset, 72.3% pass@1 and 1.08x speedup. It outperforms GPT-4 (68.1% SSE, 61.4% AVX) and Claude-3.5 (65.2% SSE, 58.9% AVX) in both correctness and performance.

Ablation

Removing VecPrompt data synthesis reduces correctness by 12.4% (from 78.5% to 66.1% on SSE) and average speedup drops from 1.12x to 1.03x. Removing VecRL reduces correctness by 8.7% and speedup to 1.06x, showing both components are critical for performance alignment.

Limitations

The framework is currently evaluated only on x86 SSE/AVX subsets of SimdBench, not on ARM SVE or RISC-V. The data synthesis pipeline may not cover all complex vectorization patterns (e.g., irregular memory access). The 8B model size may limit generalization to very large or rare SIMD instruction sets.

5) Why it matters

  1. Demonstrates that RL with execution feedback can effectively align LLMs for low-level hardware code generation, a domain where correctness and performance are tightly coupled.
  2. VecPrompt's RAG-based data synthesis provides a scalable method to inject domain-specific knowledge into LLMs without manual annotation.
  3. Shows that small models (8B) can outperform large closed-source models on specialized tasks when trained with targeted data and reward design.
  4. Connects to your interest in system safety by showing how automated code generation can reduce human error in performance-critical systems.

6) Next steps

  • [ ] Evaluate AutoVecCoder on ARM SVE and RISC-V vector extensions to test cross-architecture generalization.
  • [ ] Apply VecRL framework to other low-level code generation tasks (e.g., GPU kernels, assembly optimization) to assess transferability.
  • [ ] Investigate combining VecPrompt with active learning to iteratively expand the synthetic corpus for rare or complex vectorization patterns.
  • [ ] Explore safety implications: can RL with performance reward inadvertently generate code that is fast but unsafe (e.g., memory overruns)?
