Drupal Core SQL Injection Bug Actively Exploited, Added to CISA KEV

Drupal Core SQL Injection Bug Actively Exploited, Added to CISA KEV

supply-chain-attack
启示 · Insight

For a system security researcher focused on OS/kernel and software safety, this incident highlights that SQL injection in a widely-deployed content management system can lead to privilege escalation and remote code execution, effectively bypassing application-layer defenses. The rapid exploitation (within two days of patch release) underscores the critical need for automated patch management and the value of fuzzing database abstraction layers in large codebases. The attack pattern—probing followed by potential data extraction—mirrors common exploitation lifecycles seen in memory-corruption bugs, but here the entry point is a web-facing API rather than a kernel interface.

对于专注于操作系统/内核和软件安全的系统安全研究人员来说，此事件突显了广泛部署的内容管理系统中的 SQL 注入可导致权限提升和远程代码执行，有效绕过应用层防御。补丁发布后两天内即出现快速利用，凸显了自动化补丁管理的必要性，以及在大型代码库中对数据库抽象层进行模糊测试的价值。攻击模式——探测后可能进行数据提取——与内存损坏漏洞中常见的利用生命周期相似，但这里的入口点是面向 Web 的 API，而非内核接口。

Summary

CISA added CVE-2026-9082, a critical SQL injection vulnerability in Drupal Core, to its KEV catalog after active exploitation was observed, with over 15,000 attack attempts targeting 6,000 sites across 65 countries.

摘要

CISA 将 Drupal Core 中的严重 SQL 注入漏洞 CVE-2026-9082 添加到其 KEV 目录，原因是已观察到活跃利用，超过 15,000 次攻击尝试针对 65 个国家的 6,000 个站点。

Deep Note

Key Points

  • CVE-2026-9082 is an SQL injection vulnerability in Drupal Core with a CVSS score of 6.5.
  • It allows privilege escalation and remote code execution via specially crafted requests to the database abstraction API.
  • CISA added it to the Known Exploited Vulnerabilities (KEV) catalog based on active exploitation evidence.
  • Patches were released for multiple Drupal versions (11.3.10, 11.2.12, 11.1.10, 10.6.9, 10.5.10, 10.4.10, 9.5, 8.9).
  • Imperva observed over 15,000 attack attempts targeting nearly 6,000 sites across 65 countries.
  • Attacks primarily target gaming and financial services sites (almost 50% of attacks).
  • Most observed activity is probing/reconnaissance, but successful exploitation could lead to data extraction or privilege escalation.
  • FCEB agencies must apply fixes by May 27, 2026.

Context & Contrast

SQL injection vulnerabilities in CMS platforms are well-known, but this case is notable for the speed of exploitation (within 48 hours of patch release) and the direct linkage to privilege escalation and RCE, not just data exfiltration. Unlike typical memory-corruption bugs in kernels or runtimes, this flaw resides in the application's database abstraction layer, yet its impact (RCE, privilege escalation) overlaps with classic system-level vulnerabilities. The inclusion in CISA KEV and the scale of probing (15,000 attempts) indicate a shift toward weaponizing SQLi for full system compromise rather than simple data theft.

Why It Matters

This vulnerability bridges the gap between web application security and system security: an SQL injection in a CMS can lead to full system compromise (RCE, privilege escalation). For researchers studying exploitation techniques, this demonstrates that database abstraction APIs are a viable attack surface for gaining code execution, similar to how kernel syscalls or driver interfaces are exploited. The rapid exploitation timeline reinforces the importance of proactive patch deployment and the need for runtime defenses (e.g., WAF, RASP) that can detect SQLi-based RCE attempts. The targeting of PostgreSQL-backed configurations suggests that database-specific features (e.g., stored procedures, extensions) may be leveraged for privilege escalation.

Critical Take

The article relies heavily on vendor reports (Imperva, Drupal) and CISA announcements; independent verification of the exploitation method and the exact RCE/privilege escalation mechanism is not provided. The CVSS score of 6.5 (medium severity) seems low given the claimed impact (RCE, privilege escalation), which may indicate that the attack requires specific conditions (e.g., PostgreSQL backend, specific API usage). The 'active exploitation' evidence is based on observed probing, not confirmed successful compromises. The attribution to 'attackers' is vague; no specific threat actor or campaign is identified.

What to Watch

  • Detailed technical analysis or PoC for CVE-2026-9082, especially how the SQL injection leads to RCE.
  • Whether the vulnerability is exploited in PostgreSQL-only setups or also affects MySQL/SQLite.
  • Emergence of exploit kits or botnets incorporating this vulnerability.
  • Patch adoption rates and reports of successful data breaches or ransomware deployments linked to this CVE.
  • Updates from Drupal on whether the flaw was introduced in a specific code change (regression analysis).

背景与对比

CMS 平台中的 SQL 注入漏洞众所周知，但此案例的突出之处在于利用速度（补丁发布后 48 小时内）以及直接关联到权限提升和 RCE，而不仅仅是数据泄露。与内核或运行时中典型的内存损坏漏洞不同，此缺陷位于应用程序的数据库抽象层，但其影响（RCE、权限提升）与经典的系统级漏洞重叠。纳入 CISA KEV 以及探测规模（15,000 次尝试）表明，攻击者正将 SQLi 武器化以实现完全系统入侵，而非简单的数据窃取。

重要性

此漏洞弥合了 Web 应用安全与系统安全之间的差距：CMS 中的 SQL 注入可导致完全系统入侵（RCE、权限提升）。对于研究利用技术的研究人员来说，这表明数据库抽象 API 是获取代码执行能力的可行攻击面，类似于内核系统调用或驱动程序接口被利用的方式。快速的利用时间线强化了主动部署补丁的重要性，以及需要能够检测基于 SQLi 的 RCE 尝试的运行时防御（例如 WAF、RASP）。针对 PostgreSQL 配置的攻击表明，数据库特定功能（例如存储过程、扩展）可能被用于权限提升。

批判性视角

文章严重依赖供应商报告（Imperva、Drupal）和 CISA 公告；未提供对利用方法及确切 RCE/权限提升机制的独立验证。考虑到声称的影响（RCE、权限提升），CVSS 评分 6.5（中等严重性）似乎偏低，这可能表明攻击需要特定条件（例如 PostgreSQL 后端、特定 API 使用）。“活跃利用”的证据基于观察到的探测，而非确认的成功入侵。对“攻击者”的归因模糊；未识别出特定的威胁行为者或活动。
