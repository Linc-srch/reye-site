Megalodon GitHub Attack Targets 5,561 Repos with Malicious CI/CD Workflows

Megalodon GitHub Attack Targets 5,561 Repos with Malicious CI/CD Workflows

supply-chain-attack
启示 · Insight

For a system security researcher focused on software supply-chain security and vulnerability exploitation, this campaign demonstrates a new, highly automated vector for compromising the software supply chain at scale. It bypasses traditional code-review gates by targeting the CI/CD pipeline itself, turning the build infrastructure into a weapon. The use of forged author identities and throwaway accounts highlights the inadequacy of current identity-based trust models in open-source ecosystems. The exfiltration of OIDC tokens and cloud metadata credentials (IMDS) is particularly critical, as it enables lateral movement into cloud environments, expanding the blast radius beyond the repository.

对于专注于软件供应链安全和漏洞利用的系统安全研究员而言，此活动展示了一种新型、高度自动化的规模化软件供应链攻击向量。它通过直接攻击 CI/CD 管道本身，绕过了传统的代码审查关卡，将构建基础设施武器化。使用伪造的作者身份和一次性账户突显了当前开源生态系统中基于身份的信任模型的不足。窃取 OIDC 令牌和云元数据凭证（IMDS）尤为关键，因为它能实现向云环境的横向移动，将爆炸半径扩展到仓库之外。

Summary

A large-scale automated campaign named 'Megalodon' compromised 5,561 GitHub repositories in six hours by injecting malicious CI/CD workflows that exfiltrate credentials and secrets, linked to the broader TeamPCP supply chain attack wave.

摘要

名为 'Megalodon' 的大规模自动化攻击活动在六小时内入侵了 5,561 个 GitHub 仓库，通过注入恶意 CI/CD 工作流来窃取凭证和密钥，与更广泛的 TeamPCP 供应链攻击浪潮相关。

Deep Note

Key Points

  • 5,718 malicious commits pushed to 5,561 distinct GitHub repositories within a 6-hour window.
  • Attackers used throwaway GitHub accounts and forged author identities (build-bot, auto-ci, etc.) to inject malicious GitHub Actions workflows.
  • Payloads are base64-encoded bash scripts that exfiltrate CI secrets, cloud credentials (AWS, GCP, Azure), SSH keys, OIDC tokens, and source code secrets to a C2 server.
  • Two payload variants: 'SysDiag' (mass, triggers on push/PR) and 'Optimize-Build' (targeted, triggers on workflow_dispatch).
  • The campaign is attributed to the TeamPCP group, which has also compromised TanStack, Grafana, OpenAI, and Mistral AI.
  • The attack exploits compromised Personal Access Tokens (PATs) or deploy keys, bypassing 2FA.
  • NPM responded by invalidating granular access tokens that bypass 2FA, urging a switch to Trusted Publishing.

Context & Contrast

This attack represents a significant escalation from previous supply chain attacks (e.g., SolarWinds, Codecov) by being fully automated, massively parallel (5,500+ repos in hours), and specifically targeting the CI/CD pipeline as the primary infection vector rather than just source code. It contrasts with earlier dependency confusion or typosquatting attacks by compromising the build process itself. The scale and speed are unprecedented, and the use of OIDC token theft to pivot into cloud infrastructure is a more advanced post-exploitation technique than seen in typical open-source malware campaigns.

Why It Matters

This campaign demonstrates a critical shift in software supply chain attacks: the CI/CD pipeline is now a primary target, not just a delivery mechanism. The ability to compromise thousands of repositories in hours means that the window for detection and response is extremely narrow. For system security, this highlights the need for stronger isolation of CI/CD runners (e.g., using ephemeral, network-restricted environments), mandatory code signing for workflow changes, and runtime monitoring of pipeline behavior (e.g., detecting unexpected outbound connections or access to cloud metadata endpoints). The attack also underscores the fragility of the current PAT/deploy key model and the urgent need for more robust, short-lived credential mechanisms like OIDC with strict audience claims.

Critical Take

While the scale is alarming, the attack's success depends on already-compromised PATs or deploy keys. The root cause is credential hygiene, not a novel vulnerability in GitHub Actions itself. The report does not detail how the initial PATs were compromised (e.g., phishing, leaked in public repos, malware on developer machines). The attribution to TeamPCP is based on behavioral links, not definitive technical evidence. The effectiveness of npm's token invalidation is a temporary measure; the underlying problem of developers storing long-lived, powerful tokens remains.

What to Watch

  • Follow-up research on the initial compromise vector for the PATs/deploy keys.
  • Development of runtime security tools for CI/CD pipelines that can detect anomalous behavior (e.g., unexpected secret exfiltration, access to IMDS).
  • Adoption rate of short-lived credentials (e.g., OIDC with audience restrictions) vs. long-lived PATs in the open-source ecosystem.
  • Any new CVEs or security advisories related to GitHub Actions runner isolation or workflow integrity.
  • Evolution of TeamPCP's tactics, particularly any shift towards exploiting other CI/CD platforms (GitLab CI, CircleCI).
深度笔记

要点

  • 在 6 小时内向 5,561 个不同的 GitHub 仓库推送了 5,718 个恶意提交。
  • 攻击者使用一次性 GitHub 账户和伪造的作者身份（如 build-bot, auto-ci）注入恶意的 GitHub Actions 工作流。
  • 有效载荷是 Base64 编码的 bash 脚本，用于将 CI 密钥、云凭证（AWS、GCP、Azure）、SSH 密钥、OIDC 令牌和源代码密钥外泄到 C2 服务器。
  • 两种有效载荷变体：'SysDiag'（大规模，在 push/PR 时触发）和 'Optimize-Build'（针对性，在 workflow_dispatch 时触发）。
  • 该活动归因于 TeamPCP 组织，该组织此前还入侵了 TanStack、Grafana、OpenAI 和 Mistral AI。
  • 攻击利用了泄露的个人访问令牌（PAT）或部署密钥，绕过了双因素认证（2FA）。
  • npm 的回应是使绕过 2FA 的细粒度访问令牌失效，并敦促用户切换到 Trusted Publishing。

背景与对比

与之前的供应链攻击（如 SolarWinds、Codecov）相比，此次攻击代表了显著的升级：它是完全自动化的、大规模并行的（数小时内攻击 5500+ 仓库），并且专门针对 CI/CD 管道作为主要感染向量，而不仅仅是源代码。它与早期的依赖混淆或域名抢注攻击不同，因为它破坏了构建过程本身。其规模和速度是前所未有的，而利用窃取的 OIDC 令牌进入云基础设施是一种比典型开源恶意软件活动中所见更高级的后利用技术。

重要性

此活动展示了软件供应链攻击的一个关键转变：CI/CD 管道现在成为主要目标，而不仅仅是交付机制。能够在数小时内入侵数千个仓库意味着检测和响应的窗口极窄。对于系统安全而言，这凸显了需要更强的 CI/CD 运行器隔离（例如，使用临时的、网络受限的环境）、对工作流变更进行强制代码签名，以及对管道行为进行运行时监控（例如，检测意外的出站连接或对云元数据端点的访问）。该攻击还强调了当前 PAT/部署密钥模型的脆弱性，以及迫切需要更健壮、短期的凭证机制，例如具有严格 audience 声明的 OIDC。

批判性视角

虽然规模惊人，但该攻击的成功依赖于已被泄露的 PAT 或部署密钥。根本原因是凭证卫生问题，而非 GitHub Actions 本身存在新型漏洞。报告未详细说明最初的 PAT 是如何被泄露的（例如，网络钓鱼、在公共仓库中泄露、开发者机器上的恶意软件）。将攻击归因于 TeamPCP 是基于行为关联，而非确凿的技术证据。npm 的令牌失效措施是临时性的；开发者存储长期有效、高权限令牌的根本问题依然存在。

后续关注

  • 对 PAT/部署密钥初始入侵向量的后续研究。
  • 针对 CI/CD 管道的运行时安全工具的开发，能够检测异常行为（例如，意外的密钥外泄、访问 IMDS）。
  • 开源生态系统中短生命周期凭证（例如，带有 audience 限制的 OIDC）相对于长生命周期 PAT 的采用率。
  • 任何与 GitHub Actions 运行器隔离或工作流完整性相关的新 CVE 或安全公告。
  • TeamPCP 策略的演变，特别是是否转向利用其他 CI/CD 平台（GitLab CI、CircleCI）。
