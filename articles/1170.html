PEEK: Context Map as an Orientation Cache for Long-Context LLM Agents

PEEK：面向长上下文LLM智能体的上下文地图定向缓存

Zhuohan Gu, Qizheng Zhang, Omar Khattab, Samuel Madden · 2026-05-20 · arxiv

Summary

PEEK introduces a context map that caches orientation knowledge about a recurring external context, enabling LLM agents to achieve higher accuracy and efficiency on long-context tasks with significantly fewer iterations and lower cost.

摘要

本文提出PEEK，通过一个恒定大小的上下文地图缓存可复用的定向知识（如内容组织、有用实体），使LLM智能体在重复处理相同外部上下文时显著提升准确率和效率。实验表明，PEEK在长上下文推理任务上提升6.3-34.0%，同时迭代次数减少93-145次，成本降低1.7-5.8倍。

Deep Note

Deep Note — peek

Key Takeaways - PEEK caches reusable orientation knowledge (e.g., content organization, useful entities) about a recurring external context as a small, constant-sized context map in the agent's prompt. - The context map is maintained by a programmable cache policy with three modules: Distiller, Cartographer, and Evictor. - PEEK improves long-context reasoning (OOLONG) by 6.3-34.0% and context learning (CL-bench) by 6.0-14.0% solving rate and 7.8-12.1% rubric accuracy. - PEEK uses 93-145 fewer iterations and costs 1.7-5.8x less than the SOTA prompt-learning framework ACE on OOLONG. - Gains generalize across LMs (GPT-5.5, Qwen3-Coder-Next-FP8) and agent architectures (OpenAI Codex).

1) Why-read

PEEK introduces a context map that caches orientation knowledge about a recurring external context, enabling LLM agents to achieve higher accuracy and efficiency on long-context tasks with significantly fewer iterations and lower cost.

2) CRGP

C — Context

LLM agents increasingly operate over long and recurring external contexts (e.g., document corpora, code repositories). Existing methods preserve either the agent's trajectory, passive access to raw material, or task-level strategies, but none preserves reusable orientation knowledge about the context itself.

R — Related work

  • Shared chat and history compaction preserve agent trajectory but discard reusable context knowledge.
  • RAG, context offloading, and context compaction provide passive access to raw or condensed material but do not maintain a curated artifact about the context.
  • Prompt learning (e.g., ACE) curates task-level strategies but not contextual knowledge about a recurring external context.
  • KV-cache management is a model-level optimization, not an agent-level artifact for curated meaning.

G — Gap

Existing methods for long-context LLM agents preserve either the agent's trajectory, passive access to raw material, or task-level strategies. None preserves reusable orientation knowledge (e.g., what the context contains, how it is organized, useful entities/schemas) about the recurring context itself, which is most needed for repeated same-context workloads.

P — Proposal

PEEK introduces a context map: a small, constant-sized artifact in the agent's prompt that stores reusable orientation knowledge about the external context. The map is maintained by a programmable cache policy with three modules: a Distiller that extracts transferable knowledge from inference-time signals, a Cartographer that translates it into structured edits, and a priority-based Evictor that enforces a fixed token budget.

3) Figures

4) Experiments

Main results

On OOLONG (long-context reasoning/aggregation), PEEK improves over baselines by 6.3-34.0%, uses 93-145 fewer iterations than ACE, and costs 1.7-5.8x less than ACE. On CL-bench (context learning), PEEK improves solving rate by 6.0-14.0% and rubric accuracy by 7.8-12.1% at 1.4x lower cost than ACE. Gains hold with GPT-5.5, Qwen3-Coder-Next-FP8, and OpenAI Codex.

Ablation

Separating extraction (Distiller) from editing (Cartographer) is essential: without it, non-contextual facts leak into the map and updates become noisy and duplicative (Section 4.5).

Limitations

['The context map is designed for recurring same-context workloads; it may not benefit one-shot or rapidly changing contexts.', 'The system assumes the external context is static or slowly evolving; frequent context changes may require costly map re-initialization.', "The map's fixed token budget may limit the amount of orientation knowledge that can be stored, potentially missing rare but important details."]

5) Why it matters

  1. PEEK directly addresses the challenge of repeated interaction with large, static corpora, which is common in enterprise data analysis and code repository management.
  2. The concept of a context map as a persistent, curated artifact in the prompt offers a new paradigm for agent memory that is orthogonal to trajectory or task-level caching.
  3. The significant cost and iteration reductions (1.4-5.8x lower cost, 93-145 fewer iterations) make PEEK practical for production deployments where API costs and latency matter.

PEEK通过缓存关于重复外部上下文的定向知识，使LLM智能体在长上下文任务中以更少的迭代和更低的成本实现更高的准确率和效率。

2) CRGP（背景 / 相关工作 / 差距 / 方案）

上下文：LLM智能体越来越多地处理长且重复的外部上下文（如文档语料库、代码仓库）。现有方法要么保存智能体的轨迹，要么提供对原始材料的被动访问，要么保存任务级策略，但没有一种方法保存关于上下文本身的可复用定向知识。

相关工作：共享聊天和历史压缩保存智能体轨迹但丢弃可复用的上下文知识；RAG、上下文卸载和上下文压缩提供对原始或压缩材料的被动访问，但不维护关于上下文的精心策划的工件；提示学习（如ACE）策划任务级策略，但不保存关于重复外部上下文的上下文知识；KV缓存管理是模型级优化，而非用于策划意义的智能体级工件。

差距：现有针对长上下文LLM智能体的方法要么保存智能体轨迹，要么提供对原始材料的被动访问，要么保存任务级策略。没有一种方法保存关于重复上下文本身的可复用定向知识（例如上下文包含什么、如何组织、有用的实体/模式），而这正是重复相同上下文工作负载最需要的。

提案：PEEK引入上下文地图：智能体提示中一个恒定大小的工件，存储关于外部上下文的可复用定向知识。该地图由可编程缓存策略维护，包含三个模块：Distiller（从推理时信号中提取可迁移知识）、Cartographer（将其转化为结构化编辑）和基于优先级的Evictor（强制执行固定令牌预算）。

3) 实验

在OOLONG（长上下文推理/聚合）上，PEEK相比基线提升6.3-34.0%，比ACE迭代次数减少93-145次，成本降低1.7-5.8倍。在CL-bench（上下文学习）上，PEEK解决率提升6.0-14.0%，评分准确率提升7.8-12.1%，成本比ACE低1.4倍。性能提升在GPT-5.5、Qwen3-Coder-Next-FP8和OpenAI Codex上均成立。

消融实验

将提取（Distiller）与编辑（Cartographer）分离至关重要：否则非上下文事实会泄露到地图中，更新变得嘈杂且重复（第4.5节）。

局限性

['上下文地图专为重复的相同上下文工作负载设计，可能不适用于一次性或快速变化的上下文。', '系统假设外部上下文是静态或缓慢演变的；频繁的上下文变化可能需要昂贵的地图重新初始化。', '地图的固定令牌预算可能限制可存储的定向知识量，可能遗漏罕见但重要的细节。']

4) 重要性

  1. PEEK直接解决了与大型静态语料库重复交互的挑战，这在企业数据分析和代码仓库管理中很常见。
  2. 上下文地图作为提示中持久化、精心策划的工件的概念，为智能体记忆提供了一种新的范式，与轨迹或任务级缓存正交。
  3. 显著的成本和迭代减少（成本降低1.4-5.8倍，迭代减少93-145次）使PEEK在API成本和延迟重要的生产部署中具有实用性。
