TrapDoor Supply Chain Attack Spreads Credential-Stealing Malware via npm, PyPI, and CratesIO

TrapDoor Supply Chain Attack Spreads Credential-Stealing Malware via npm, PyPI, and CratesIO

content-management supply-chain-attack zero-trust
启示 · Insight

For a system security researcher focused on software supply-chain security and exploitation, TrapDoor is a significant case study in cross-ecosystem compromise. It demonstrates the evolution from simple typosquatting to multi-stage attacks that leverage ecosystem-specific execution hooks (postinstall, build.rs, import-time), credential validation against cloud APIs (AWS/GitHub), and lateral movement via SSH. The use of AI coding tool configuration files (.cursorrules, CLAUDE.md) as a persistence and exfiltration vector is a novel technique that blurs the line between supply-chain attack and developer workstation compromise, warranting deep analysis of its detection and mitigation.

对于专注于软件供应链安全和漏洞利用的系统安全研究员来说，TrapDoor 是一个重要的跨生态系统妥协案例研究。它展示了从简单的域名抢注到多阶段攻击的演变，这些攻击利用了特定生态系统的执行钩子（postinstall、build.rs、导入时执行）、针对云 API（AWS/GitHub）的凭证验证以及通过 SSH 的横向移动。使用 AI 编码工具配置文件（.cursorrules、CLAUDE.md）作为持久化和数据窃取向量是一种新颖的技术，模糊了供应链攻击和开发者工作站妥协之间的界限，值得对其检测和缓解措施进行深入分析。

Summary

A coordinated supply chain attack named TrapDoor has been discovered across npm, PyPI, and Crates.io, deploying over 34 malicious packages designed to steal developer credentials, crypto wallets, and cloud secrets, with novel persistence mechanisms including AI-assisted exfiltration via .cursorrules and CLAUDE.md files.

摘要

一个名为 TrapDoor 的协同供应链攻击在 npm、PyPI 和 Crates.io 上被发现，部署了超过 34 个恶意包，旨在窃取开发者凭证、加密钱包和云密钥，并采用新颖的持久化机制，包括通过 .cursorrules 和 CLAUDE.md 文件进行 AI 辅助的数据窃取。

Deep Note

Key Points

  • Cross-ecosystem attack targeting npm, PyPI, and Crates.io with 34+ malicious packages across 384+ versions.
  • Targets developers in crypto, DeFi, Solana, and AI communities.
  • Uses ecosystem-specific execution: postinstall hooks (npm), build.rs (Rust crates), import-time execution (PyPI).
  • Deploys a shared JS payload (trap-core.js) that scans for credentials, validates AWS/GitHub tokens, and performs SSH lateral movement.
  • Establishes persistence via .cursorrules, CLAUDE.md, Git hooks, shell hooks, systemd, cron, and SSH.
  • Exfiltrates data via GitHub Gists using XOR-encrypted keystores.
  • Opens GitHub PRs to inject hidden instructions into AI project files (langchain, browser-use, langflow).

Context & Contrast

Prior supply-chain attacks (e.g., SolarWinds, Codecov, event-stream) typically focused on a single ecosystem or used generic backdoors. TrapDoor is notable for its coordinated multi-ecosystem approach (npm, PyPI, Crates.io) and its novel use of AI coding assistant configuration files as a persistence and exfiltration channel. This represents a shift from passive package poisoning to active, multi-vector compromise of the developer environment, including lateral movement and cloud credential validation.

Why It Matters

This attack highlights the expanding attack surface of the software supply chain, now incorporating AI tooling configurations. For system security researchers, it underscores the need for runtime monitoring of developer workstations, validation of package execution hooks across ecosystems, and detection of anomalous SSH lateral movement and cloud API calls from build processes. The technique of poisoning AI configuration files to exfiltrate secrets represents a new class of threat that requires updated security policies for CI/CD pipelines and developer environments.

Critical Take

The report is based on analysis by Socket, a security vendor, and while the technical details are credible, the full scope of compromise (number of affected developers, actual data exfiltration) is not independently verified. The claim of AI assistant 'trickery' via .cursorrules and CLAUDE.md is novel but the practical impact (whether any AI assistant actually executed the hidden instructions) requires further validation. The attribution to a single campaign is based on behavioral similarities; some packages may be copycat or unrelated.

What to Watch

  • Monitor for similar cross-ecosystem campaigns using AI configuration files as attack vectors.
  • Track GitHub PRs targeting popular AI/developer projects for hidden instructions.
  • Watch for updates from npm, PyPI, and Crates.io security teams on package takedowns and ecosystem-wide mitigations.
  • Observe if threat actors adopt similar techniques for other package managers (RubyGems, Go modules).
  • Look for research on detecting and sandboxing build.rs and postinstall hooks at scale.
深度笔记

要点

  • 跨生态系统攻击，针对 npm、PyPI 和 Crates.io，涉及超过 34 个恶意包，共 384 多个版本。
  • 针对加密、DeFi、Solana 和 AI 社区的开发者。
  • 利用特定生态系统的执行方式：npm 的 postinstall 钩子、Rust crate 的 build.rs、PyPI 的导入时执行。
  • 部署共享的 JS 载荷（trap-core.js），用于扫描凭证、验证 AWS/GitHub 令牌并执行 SSH 横向移动。
  • 通过 .cursorrules、CLAUDE.md、Git 钩子、shell 钩子、systemd、cron 和 SSH 建立持久化。
  • 使用 XOR 加密的密钥库通过 GitHub Gist 窃取数据。
  • 通过 GitHub PR 向 AI 项目文件（langchain、browser-use、langflow）注入隐藏指令。

背景与对比

以往的供应链攻击（例如 SolarWinds、Codecov、event-stream）通常专注于单一生态系统或使用通用后门。TrapDoor 以其协调的多生态系统方法（npm、PyPI、Crates.io）以及将 AI 编码助手配置文件用作持久化和数据窃取渠道的新颖方式而引人注目。这代表了从被动的包投毒向主动、多向量入侵开发者环境的转变，包括横向移动和云凭证验证。

重要性

这次攻击凸显了软件供应链攻击面的扩大，现在已纳入 AI 工具配置。对于系统安全研究员来说，它强调了需要对开发者工作站进行运行时监控、跨生态系统验证包执行钩子，以及检测来自构建过程的异常 SSH 横向移动和云 API 调用。投毒 AI 配置文件以窃取机密的技术代表了一类新的威胁，需要为 CI/CD 流水线和开发者环境更新安全策略。

批判性视角

该报告基于安全供应商 Socket 的分析，虽然技术细节可信，但妥协的全部范围（受影响的开发者数量、实际数据窃取）尚未得到独立验证。通过 .cursorrules 和 CLAUDE.md 进行 AI 助手‘欺骗’的说法很新颖，但其实际影响（是否有任何 AI 助手实际执行了隐藏指令）需要进一步验证。将攻击归因于单一活动是基于行为相似性；某些包可能是模仿或无关的。

后续关注

  • 关注使用 AI 配置文件作为攻击向量的类似跨生态系统活动。
  • 追踪针对流行 AI/开发者项目的 GitHub PR，查找隐藏指令。
  • 关注 npm、PyPI 和 Crates.io 安全团队关于包下架和生态系统范围缓解措施的更新。
  • 观察威胁行为者是否对其他包管理器（RubyGems、Go modules）采用类似技术。
  • 寻找关于大规模检测和沙箱化 build.rs 和 postinstall 钩子的研究。
