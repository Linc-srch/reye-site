TrapDoor Supply Chain Attack Spreads Credential-Stealing Malware via npm, PyPI, and CratesIO

TrapDoor Supply Chain Attack Spreads Credential-Stealing Malware via npm, PyPI, and CratesIO

content-management supply-chain-attack zero-trust
启示 · Insight

For a system security researcher, this campaign is a critical case study in the evolution of software supply chain attacks. It demonstrates a multi-ecosystem (npm, PyPI, Crates.io) approach that bypasses single-registry defenses, uses ecosystem-specific execution vectors (postinstall hooks, build.rs, import-time execution), and introduces a novel persistence and data exfiltration technique by poisoning AI coding assistant configuration files (.cursorrules, CLAUDE.md). This blurs the line between supply chain compromise and AI-assisted social engineering, creating a new attack surface for developer workstations that traditional endpoint security may not cover.

对于系统安全研究员来说，这次活动是软件供应链攻击演变中的一个关键案例研究。它展示了一种跨生态系统（npm、PyPI、Crates.io）的攻击方法，绕过了单一注册表的防御，利用了特定生态系统的执行向量（postinstall 钩子、build.rs、导入时执行），并通过投毒 AI 编码助手的配置文件（.cursorrules、CLAUDE.md）引入了一种新颖的持久化和数据窃取技术。这模糊了供应链妥协与 AI 辅助社会工程之间的界限，为开发者工作站创造了一个传统端点安全可能无法覆盖的新攻击面。

Summary

A coordinated supply chain attack named TrapDoor has been discovered across npm, PyPI, and Crates.io, deploying over 34 malicious packages designed to steal credentials, crypto wallets, and developer secrets, with novel persistence mechanisms including AI-assisted exfiltration via .cursorrules and CLAUDE.md files.

摘要

一场名为 TrapDoor 的协同供应链攻击在 npm、PyPI 和 Crates.io 上被发现，部署了超过 34 个恶意包，旨在窃取凭证、加密钱包和开发者机密，并采用了通过 .cursorrules 和 CLAUDE.md 文件进行 AI 辅助数据窃取的新型持久化机制。

Deep Note

Key Points

  • Multi-ecosystem attack targeting npm, PyPI, and Crates.io with 34+ malicious packages across 384 versions.
  • Malware steals developer secrets, crypto wallets, SSH keys, cloud credentials, and environment variables.
  • Uses ecosystem-specific execution: npm postinstall hooks, Rust build.rs scripts, Python import-time execution.
  • Novel persistence via .cursorrules and CLAUDE.md files to trick AI coding assistants into executing exfiltration commands.
  • Lateral movement via SSH and persistence via cron, systemd, Git hooks, and shell hooks.
  • Remote payload hosted on GitHub Pages for dynamic updates without re-publishing packages.
  • Targeted at crypto, DeFi, Solana, and AI developer communities.

Context & Contrast

This attack is more sophisticated than typical typosquatting or dependency confusion campaigns. It combines multiple execution vectors across three major package registries, which is rare. The use of AI assistant configuration files (.cursorrules, CLAUDE.md) as a persistence and exfiltration mechanism is novel and represents a significant evolution from traditional supply chain attacks that only rely on post-install scripts or runtime code injection. It also demonstrates a shift from purely technical exploitation to manipulating AI-assisted development workflows.

Why It Matters

This campaign highlights a critical blind spot in current software supply chain security: the lack of defense-in-depth for developer workstations against attacks that leverage AI coding tools. The ability to poison AI assistant configurations via pull requests on popular open-source projects creates a new, hard-to-detect attack vector. For system security, it underscores the need for runtime monitoring of AI tool behavior, stricter sandboxing of package installation scripts, and validation of configuration files that influence code generation and execution.

Critical Take

The analysis is based on a single vendor's (Socket) report. The full scope of compromise, including the number of affected developers and the success rate of the AI assistant poisoning via PRs, is unverified. The claim that the AI assistants actually executed the hidden instructions in .cursorrules/CLAUDE.md needs independent confirmation. The attribution to a single threat actor is based on behavioral clustering, which may be incomplete.

What to Watch

  • Independent verification of the AI assistant poisoning technique (e.g., by other security researchers or AI vendors).
  • Disclosure of the specific GitHub PRs used to inject .cursorrules/CLAUDE.md files.
  • Response from GitHub, npm, PyPI, and Crates.io regarding package takedowns and account bans.
  • Emergence of similar attacks targeting other AI coding tools (e.g., GitHub Copilot, Amazon CodeWhisperer).
  • Development of detection rules for malicious .cursorrules and CLAUDE.md content in CI/CD pipelines.
深度笔记

要点

  • 跨生态系统攻击，针对 npm、PyPI 和 Crates.io，包含 34 多个恶意包，共 384 个版本。
  • 恶意软件窃取开发者机密、加密钱包、SSH 密钥、云凭证和环境变量。
  • 使用特定生态系统执行方式：npm 的 postinstall 钩子、Rust 的 build.rs 脚本、Python 的导入时执行。
  • 通过 .cursorrules 和 CLAUDE.md 文件实现新型持久化，诱骗 AI 编码助手执行数据窃取命令。
  • 通过 SSH 进行横向移动，并通过 cron、systemd、Git 钩子和 shell 钩子实现持久化。
  • 远程载荷托管在 GitHub Pages 上，无需重新发布包即可动态更新。
  • 针对加密货币、DeFi、Solana 和 AI 开发者社区。

背景与对比

这次攻击比典型的域名抢注或依赖混淆活动更为复杂。它结合了三个主要包注册表上的多种执行向量，这很罕见。使用 AI 助手配置文件（.cursorrules、CLAUDE.md）作为持久化和数据窃取机制是新颖的，代表了从仅依赖安装后脚本或运行时代码注入的传统供应链攻击的重大演变。它还展示了从纯粹的技术利用向操纵 AI 辅助开发工作流程的转变。

重要性

这次活动凸显了当前软件供应链安全中的一个关键盲点：缺乏针对利用 AI 编码工具的攻击的开发者工作站深度防御。通过在流行开源项目上发起拉取请求来投毒 AI 助手配置的能力，创造了一个新的、难以检测的攻击向量。对于系统安全而言，它强调了需要对 AI 工具行为进行运行时监控、对包安装脚本进行更严格的沙箱化，以及对影响代码生成和执行的配置文件进行验证。

批判性视角

该分析基于单一供应商（Socket）的报告。受影响开发者的数量以及通过 PR 进行 AI 助手投毒的成功率等全部妥协范围尚未得到验证。关于 AI 助手实际执行了 .cursorrules/CLAUDE.md 中隐藏指令的说法需要独立确认。将攻击归因于单个威胁行为者是基于行为聚类，这可能不完整。

后续关注

  • 其他安全研究人员或 AI 供应商对 AI 助手投毒技术的独立验证。
  • 用于注入 .cursorrules/CLAUDE.md 文件的具体 GitHub PR 的披露。
  • GitHub、npm、PyPI 和 Crates.io 关于下架包和封禁账户的回应。
  • 针对其他 AI 编码工具（如 GitHub Copilot、Amazon CodeWhisperer）的类似攻击的出现。
  • 在 CI/CD 管道中针对恶意 .cursorrules 和 CLAUDE.md 内容的检测规则的开发。
