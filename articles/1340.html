Malicious npm Package Stole Files From Claude AI User Directory via GitHub

Malicious npm Package Stole Files From Claude AI User Directory via GitHub

content-management supply-chain-attack zero-trust
启示 · Insight

For a system security researcher focused on software supply-chain security, this incident is a textbook example of a low-sophistication, high-impact supply-chain attack targeting the npm ecosystem. The malware's use of a hardcoded GitHub token and its sloppy OPSEC (leaking the token) highlights that the barrier to entry for creating functional, data-stealing malware has dropped dramatically, likely due to AI-assisted coding. This shifts the threat landscape: we now face a higher volume of 'sloppy' but effective malware that exploits trusted distribution channels (npm) and developer workflows (postinstall scripts). The attack also demonstrates a novel targeting vector—AI application user data directories—which expands the attack surface for supply-chain compromises.

对于专注于软件供应链安全的系统安全研究员来说，此事件是 npm 生态系统中一次低技术门槛、高影响力的供应链攻击的典型案例。恶意软件使用了硬编码的 GitHub token 且操作安全（OPSEC）极差（泄露了 token），这突显了创建功能性数据窃取恶意软件的准入门槛已大幅降低，很可能是由于 AI 辅助编码。这改变了威胁格局：我们现在面临大量‘粗糙’但有效的恶意软件，它们利用受信任的分发渠道（npm）和开发者工作流（postinstall 脚本）。该攻击还展示了一个新的攻击向量——AI 应用用户数据目录——这扩大了供应链攻击的攻击面。

Summary

A malicious npm package named 'mouse5212-super-formatter' was discovered exfiltrating files from the Claude AI user data directory (/mnt/user-data) to a threat actor-controlled GitHub repository during its postinstall hook.

摘要

一个名为 'mouse5212-super-formatter' 的恶意 npm 包被发现利用 postinstall 钩子，将 Claude AI 用户数据目录 (/mnt/user-data) 中的文件窃取到攻击者控制的 GitHub 仓库。

Deep Note

Key Points

  • Package name: mouse5212-super-formatter.
  • Targets: /mnt/user-data directory (Claude AI user data).
  • Mechanism: Uses npm postinstall script to authenticate to GitHub (hardcoded or env token), creates a repo, and recursively uploads all files.
  • Estimated downloads: 676.
  • Attacker OPSEC failure: Hardcoded GitHub token leaked in the package, suggesting AI-generated malware.
  • GitHub account created hours before the first malicious version was uploaded.

Context & Contrast

This attack is a continuation of the long-standing trend of malicious npm packages using postinstall scripts for data exfiltration (e.g., earlier typosquatting or dependency confusion attacks). However, it is novel in its specific targeting of an AI application's local data directory, reflecting the growing value of AI-related data. The sloppy OPSEC (leaked token) contrasts with more sophisticated, stealthy supply-chain attacks that use obfuscation or multi-stage payloads. This suggests a new class of 'AI-augmented script kiddie' threat actors who can produce functional malware but lack operational security discipline.

Why It Matters

This incident underscores a critical vulnerability in the software supply chain: the npm registry remains a viable vector for data theft, and the barrier to entry for creating such malware is now negligible. For system security, it highlights that developer machines and CI/CD pipelines are prime targets, and that AI application data (e.g., Claude user files) is a high-value asset. The sloppy OPSEC is a double-edged sword: it makes attribution easier but also signals a flood of low-quality, high-volume attacks that will overwhelm traditional detection mechanisms. It also validates the need for runtime monitoring of postinstall scripts and stricter npm package validation.

Critical Take

The report's claim that the malware is 'AI-generated' is circumstantial (based on OPSEC failure) and not definitively proven. The actual impact (how many real installs vs. downloads) is unclear. The attack is technically trivial—a simple file upload script—and does not represent a novel exploitation technique. The hype around 'AI slop' may overshadow the more fundamental issue: the npm registry's lack of effective pre-publication scanning for malicious postinstall behavior. The hardcoded token leak is a significant attribution signal, but the attacker's identity remains unknown.

What to Watch

  • Monitor npm for similar packages targeting AI application directories (e.g., ChatGPT, Copilot local data).
  • Watch for increased volume of npm packages with suspicious postinstall scripts that authenticate to external services.
  • Track if npm registry implements automated scanning for hardcoded tokens or suspicious network calls in postinstall hooks.
  • Observe if threat actors improve OPSEC in future AI-generated malware, making detection harder.
深度笔记

要点

  • 包名：mouse5212-super-formatter。
  • 目标：/mnt/user-data 目录（Claude AI 用户数据）。
  • 机制：利用 npm postinstall 脚本向 GitHub 进行身份验证（使用硬编码或环境变量中的 token），创建仓库，并递归上传所有文件。
  • 估计下载量：676 次。
  • 攻击者 OPSEC 失败：包中泄露了硬编码的 GitHub token，暗示恶意软件由 AI 生成。
  • GitHub 账户在首个恶意版本上传前几小时创建。

背景与对比

此攻击是恶意 npm 包利用 postinstall 脚本进行数据窃取这一长期趋势的延续（例如早期的域名抢注或依赖混淆攻击）。然而，其新颖之处在于专门针对 AI 应用的本地数据目录，反映了 AI 相关数据日益增长的价值。糟糕的 OPSEC（泄露的 token）与使用混淆或多阶段 payload 的更复杂、隐蔽的供应链攻击形成对比。这表明出现了一类新的‘AI 增强的脚本小子’威胁行为者，他们能够生成功能性的恶意软件，但缺乏操作安全纪律。

重要性

此事件强调了软件供应链中的一个关键漏洞：npm 注册表仍然是数据窃取的有效载体，而创建此类恶意软件的准入门槛现已微不足道。对于系统安全而言，它突显了开发者机器和 CI/CD 管道是主要目标，并且 AI 应用数据（例如 Claude 用户文件）是高价值资产。糟糕的 OPSEC 是一把双刃剑：它使归因更容易，但也预示着大量低质量、高数量的攻击将淹没传统的检测机制。它还验证了对 postinstall 脚本进行运行时监控以及更严格的 npm 包验证的必要性。

批判性视角

报告中关于恶意软件是‘AI 生成’的说法是间接证据（基于 OPSEC 失败），并未得到明确证实。实际影响（真实安装次数与下载次数对比）尚不清楚。该攻击在技术上微不足道——一个简单的文件上传脚本——并不代表一种新颖的利用技术。围绕‘AI 垃圾’的炒作可能会掩盖更根本的问题：npm 注册表缺乏对恶意 postinstall 行为的有效预发布扫描。硬编码的 token 泄露是一个重要的归因信号，但攻击者的身份仍然未知。

后续关注

  • 监控 npm 上针对 AI 应用目录（例如 ChatGPT、Copilot 本地数据）的类似包。
  • 关注带有可疑 postinstall 脚本（向外部服务进行身份验证）的 npm 包数量是否增加。
  • 追踪 npm 注册表是否会对 postinstall 钩子中的硬编码 token 或可疑网络调用实施自动扫描。
  • 观察威胁行为者是否会在未来的 AI 生成恶意软件中改进 OPSEC，使检测更加困难。
