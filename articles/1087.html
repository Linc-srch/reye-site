1) Why-read

This survey proposes that code serves as the executable, inspectable, and stateful medium for agent reasoning, acting, and environment modeling, shifting the focus from code as an output to code as the operational substrate for agentic AI.

2) CRGP

C — Context

Large language models (LLMs) have advanced code generation from competitive programming to repository-level software engineering. In emerging agentic systems, code is increasingly used not just as a target artifact but as a medium for reasoning, acting, and environment modeling, leading to a need for a unified framework to understand this shift.

R — Related work

  • Program-aided reasoning methods (e.g., PAL, Chain-of-Code) externalize intermediate computation into executable code.
  • Embodied agents (e.g., Code as Policies, Voyager) use generated programs as executable policies for physical or simulated worlds.
  • Software engineering agents (e.g., SWE-bench, InterCode) use codebases, execution traces, and tests as structured environment representations.
  • Agent harness frameworks (e.g., MetaHarness, AutoHarness) discuss the software layer that connects LLMs to tools, APIs, and feedback channels.

G — Gap

Existing surveys treat code primarily as the end product of LLMs, focusing on generation quality. There is no unified view that centers code as the executable and inspectable medium through which agents reason, act, maintain state, and coordinate, especially across long-horizon and multi-agent settings.

P — Proposal

The paper introduces 'code as agent harness', a unified perspective organizing agent systems into three layers: harness interface (code for reasoning, acting, environment modeling), harness mechanisms (planning, memory, tool use, feedback-driven control), and scaling the harness (multi-agent orchestration over shared code artifacts). The key insight is that code serves as the operational substrate that makes agent behavior executable, verifiable, and stateful.

4) Experiments

Main results

As a survey, the paper does not present new experimental results but summarizes representative methods across applications including coding assistants, GUI/OS automation, embodied agents, scientific discovery, personalization, DevOps, and enterprise workflows. It cites performance gains from methods like Code as Policies (robotic task success), SWE-bench (repository-level bug fixing), and Voyager (skill discovery in Minecraft).

Ablation

No ablation study is presented; the paper is a survey that synthesizes existing work and identifies open challenges.

Limitations

['The survey does not provide a quantitative comparison or benchmark for the proposed unified view.', 'The framework is conceptual and may require further empirical validation across diverse agent systems.', 'The paper focuses on code-centric harnesses, potentially underrepresenting non-code-based agent architectures (e.g., purely neural or symbolic planners).']

5) Why it matters

  1. Provides a structured taxonomy to understand how code enables closed-loop agentic behavior, relevant to building reliable AI agents.
  2. Highlights the role of agent-initiated code artifacts (e.g., tests, temporary tools, DSL programs) as a relatively underexplored area for improving agent autonomy.
  3. Connects code-centric design to key challenges like verification under incomplete feedback and multi-agent state consistency, which are critical for deploying agents in safety-critical domains.

6) Next steps

  • [ ] Explore how agent-initiated code artifacts (e.g., regression tests, reusable skills) can be automatically generated and maintained to improve long-horizon task reliability.
  • [ ] Investigate harness optimization methods that use execution feedback to iteratively improve agent code without regressing performance.
  • [ ] Design evaluation protocols that go beyond final task success to measure intermediate state consistency, failure recovery, and multi-agent coordination quality.
  • [ ] Apply the code-as-agent-harness framework to build a unified benchmark for comparing different harness architectures across domains.

Deep Note — 代码即智能体框架

主要结论 - 代码从LLM的最终输出转变为智能体运行的操作基底，使行为可执行、可验证、有状态。 - 提出三层框架：框架接口（推理/行动/环境建模）、框架机制（规划/记忆/工具/反馈控制）、框架扩展（多智能体通过共享代码工件编排）。 - 智能体自主生成的代码工件（如测试、临时工具、DSL程序）是提升自主性的重要但未被充分探索的方向。 - 该视角有助于解决不完整反馈下的验证、多智能体状态一致性等关键挑战。

1) 阅读理由

核心主张：代码应作为智能体运行的操作基底；关键观察：现有研究将代码视为输出，而本文将其视为智能体推理、行动和状态维护的统一媒介。

2) CRGP（背景 / 相关工作 / 差距 / 方案）

背景：大语言模型在代码生成方面已从竞赛编程进步到仓库级软件工程。在新兴的智能体系统中，代码不仅作为目标产物，还越来越多地被用作推理、行动和环境建模的媒介，因此需要一个统一框架来理解这一转变。

相关工作：程序辅助推理方法（如PAL、Chain-of-Code）将中间计算外化为可执行代码；具身智能体（如Code as Policies、Voyager）使用生成的程序作为物理或模拟世界的可执行策略；软件工程智能体（如SWE-bench、InterCode）将代码库、执行轨迹和测试作为结构化环境表示；智能体框架（如MetaHarness、AutoHarness）讨论了连接LLM与工具、API和反馈通道的软件层。

差距：现有综述主要将代码视为LLM的最终产物，关注生成质量。缺乏一个统一视角，将代码视为智能体推理、行动、维护状态和协调的可执行、可检查媒介，尤其是在长周期和多智能体场景中。

提议：本文引入“代码即智能体框架”这一统一视角，将智能体系统组织为三个层次：框架接口（用于推理、行动、环境建模的代码）、框架机制（规划、记忆、工具使用、反馈驱动控制）、框架扩展（通过共享代码工件进行多智能体编排）。核心洞察是代码作为操作基底，使智能体行为可执行、可验证、有状态。

3) 实验

作为一篇综述，本文未提出新的实验结果，而是总结了各类应用中的代表性方法，包括编码助手、GUI/OS自动化、具身智能体、科学发现、个性化、DevOps和企业工作流。文中引用了Code as Policies（机器人任务成功率）、SWE-bench（仓库级bug修复）和Voyager（Minecraft技能发现）等方法的性能提升。

消融实验

本文为综述论文，未进行消融实验，而是综合现有工作并识别开放挑战。

局限性

['该综述未对所提出的统一视角进行定量比较或基准测试。', '该框架是概念性的，可能需要跨不同智能体系统进行进一步实证验证。', '本文聚焦于以代码为中心的框架，可能低估了非代码基础的智能体架构（如纯神经或符号规划器）。']

4) 重要性

  1. 提供了一个结构化分类法，理解代码如何实现闭环智能体行为，对构建可靠AI智能体至关重要。
  2. 强调了智能体自主生成的代码工件（如测试、临时工具、DSL程序）作为提升智能体自主性的相对未充分探索领域。
  3. 将代码中心设计与关键挑战（如不完整反馈下的验证、多智能体状态一致性）联系起来，这些对于在安全关键领域部署智能体至关重要。

5) 下一步

  • [ ] 探索如何自动生成和维护智能体自主生成的代码工件（如回归测试、可复用技能），以提高长周期任务可靠性。
  • [ ] 研究利用执行反馈迭代改进智能体代码而不导致性能退化的框架优化方法。
  • [ ] 设计超越最终任务成功率的评估协议，衡量中间状态一致性、故障恢复和多智能体协调质量。
  • [ ] 应用“代码即智能体框架”构建跨领域的统一基准，比较不同框架架构。
