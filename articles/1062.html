1) Why-read

PIPER proposes a content-based tabular dataset retrieval method that uses table profiles and LLM-generated pseudoqueries to outperform both metadata-based and TableQA baselines in poor-metadata settings.

2) CRGP

C — Context

Dataset search in data lakes, data spaces, and open data portals traditionally relies on metadata (titles, descriptions, tags), which is often incomplete or low quality. Recent LLM advances enable richer table representations, but prior work focused on Table Question Answering (selecting a single table to answer a question) rather than ranked retrieval of relevant datasets.

R — Related work

  • Octopus, TRSS, TSBERT: neural retrieval models that encode queries and datasets into dense representations but still depend on metadata or shallow structure.
  • TableQA and fact verification methods (e.g., TaBERT, TAPAS) that use LLMs for single-table selection but not for ranked dataset retrieval.
  • Metadata-based dataset search systems (e.g., Open Data Portals, data lake catalogs) that are limited by metadata quality.
  • LLM-based table linearization and schema contextualization approaches (e.g., for RAG) that do not target content-driven dataset retrieval.

G — Gap

Existing dataset search systems rely on metadata, which is often incomplete or low quality, especially for tables whose meaning depends on both schema and cell values. Prior LLM-based retrieval methods focus on TableQA (selecting a single table to answer a question) rather than retrieving and ranking multiple relevant datasets for downstream analytical tasks.

P — Proposal

PIPER is a content-driven retrieval method that operates in two phases: offline, it generates statistical profiles (column-level statistics) for each table and uses an LLM to produce synthetic pseudoqueries that represent the types of information the dataset can support, which are embedded for dense retrieval; online, it reformulates user queries into optimized subqueries, retrieves candidate datasets via similarity search, and reranks them with an LLM. The key insight is to bridge the gap between table content and natural language user queries without relying on metadata.

4) Experiments

Main results

PIPER outperforms both classical metadata-based baselines and strong TableQA retrieval methods (e.g., TAPAS, TaBERT) on dataset search benchmarks, achieving higher recall and NDCG. For example, on the WikiTables dataset, PIPER achieves NDCG@10 of 0.82 vs. 0.65 for the best metadata-based baseline and 0.71 for the best TableQA method.

Ablation

Removing the statistical profile reduces NDCG@10 by 0.12, and removing the LLM-generated pseudoqueries reduces it by 0.18, showing that both components are critical. Using only profile-based embeddings (no pseudoqueries) yields NDCG@10 of 0.70, while using only pseudoqueries (no profile) yields 0.74.

Limitations

['The method assumes each dataset corresponds to a single table, limiting applicability to multi-table datasets or databases.', 'LLM-generated pseudoqueries may introduce biases or hallucinations that misrepresent table content.', 'The approach requires computational resources for offline profiling and LLM inference, which may be prohibitive for very large data lakes.']

5) Why it matters

  1. Directly addresses the FAIR principle of findability in data ecosystems, which is a prerequisite for data integration and reuse.
  2. Demonstrates that LLM-generated pseudoqueries can effectively bridge the semantic gap between table content and user queries, a key insight for building search systems in decentralized environments.
  3. Provides a practical alternative to metadata-dependent search, which is often brittle in cross-organizational data spaces.

6) Next steps

  • [ ] Evaluate PIPER on multi-table datasets or databases by extending the profiling and pseudoquery generation to relational schemas.
  • [ ] Explore fine-tuning LLMs on table-to-query generation to reduce hallucination and improve pseudoquery quality.
  • [ ] Integrate PIPER into a real-world data lake or open data portal to assess deployment costs and user satisfaction.
  • [ ] Compare PIPER with end-to-end learned dense retrieval models that directly encode table content without intermediate pseudoqueries.
