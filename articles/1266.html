Google API Keys Remain Active After Deletion

Google API Keys Remain Active After Deletion

content-management supply-chain-attack zero-trust
启示 · Insight

For a system security researcher focused on OS/kernel and software supply-chain security, this flaw represents a fundamental failure in access revocation and state management within a critical cloud infrastructure component. It mirrors classic TOCTOU (time-of-check time-of-use) vulnerabilities and highlights how distributed systems' eventual consistency models can create persistent security gaps. The finding underscores that cloud platforms, despite their sophistication, suffer from the same class of logic bugs found in kernels and runtimes—where a deletion operation is not immediately enforced across all replicas. This is directly relevant to supply-chain security, as leaked or compromised API keys are a primary vector for CI/CD pipeline attacks and data exfiltration.

对于专注于操作系统/内核安全和软件供应链安全的研究人员而言，此缺陷代表了关键云基础设施组件中访问撤销和状态管理的根本性失败。它类似于经典的 TOCTOU（检查时间与使用时间）漏洞，并突显了分布式系统的最终一致性模型如何造成持久的安全缺口。该发现强调，云平台尽管复杂，但仍存在与内核和运行时中相同类型的逻辑错误——即删除操作未在所有副本中立即执行。这与供应链安全直接相关，因为泄露或受损的 API 密钥是 CI/CD 流水线攻击和数据泄露的主要途径。

Summary

Researchers discovered that Google Cloud Platform API keys remain functional for authentication even after users delete them, due to distributed system caching and eventual consistency, posing a risk of unauthorized access.

摘要

研究人员发现，由于分布式系统的缓存和最终一致性模型，Google Cloud Platform 的 API 密钥在用户删除后仍可用于身份验证，存在未授权访问的风险。

Deep Note

Key Points

  • Google Cloud Platform API keys remain usable for authentication after the user has deleted them.
  • The root cause is distributed caching and eventual consistency models in the backend systems.
  • This allows an attacker who obtains a deleted key to continue using it for unauthorized access.
  • The flaw affects key lifecycle management and revocation enforcement in cloud services.

Context & Contrast

This finding is distinct from typical API key leaks (e.g., hardcoded in repos) because it reveals a systemic revocation failure rather than a user-side misconfiguration. It contrasts with prior work on cloud security that often focuses on IAM misconfigurations or privilege escalation; here, the platform itself fails to honor a user's explicit deletion command. It parallels research on 'zombie' credentials in other distributed systems (e.g., AWS IAM eventual consistency delays) but demonstrates a more severe persistence—keys remain active indefinitely rather than for a short propagation window.

Why It Matters

This is a high-severity supply-chain and infrastructure security issue. API keys are the linchpin of automated cloud operations, CI/CD pipelines, and service-to-service authentication. If deletion does not equate to immediate revocation, the entire key rotation and incident response playbook is undermined. An attacker who compromises a key and then triggers its deletion (e.g., via social engineering or a prior breach) can maintain persistent access without detection. This fundamentally breaks the security model of credential lifecycle management.

Critical Take

The paper's claims need verification regarding the exact scope and duration of key persistence. It is unclear if this affects all GCP services or only specific ones, and whether the keys eventually expire or remain active indefinitely. The researchers have not disclosed whether Google has acknowledged or patched the issue. There is a possibility that the behavior is an undocumented feature (e.g., for delayed propagation in multi-region deployments) rather than a pure bug. The practical exploitability depends on an attacker already having access to the key value.

What to Watch

  • Google's official response and any security advisory or patch.
  • Further research into similar eventual-consistency revocation flaws in AWS, Azure, and other cloud providers.
  • Proof-of-concept exploits demonstrating real-world impact (e.g., accessing GCP services with a deleted key).
  • Industry discussion on whether cloud SLAs should guarantee synchronous revocation.
  • Updates to key management tools (e.g., HashiCorp Vault) to detect or mitigate such platform-level delays.
深度笔记

要点

  • Google Cloud Platform 的 API 密钥在用户删除后仍可用于身份验证。
  • 根本原因是后端系统中的分布式缓存和最终一致性模型。
  • 这使得获取已删除密钥的攻击者能够继续使用它进行未授权访问。
  • 该缺陷影响了云服务中的密钥生命周期管理和撤销强制执行。

背景与对比

该发现不同于典型的 API 密钥泄露（例如硬编码在仓库中），因为它揭示的是系统性的撤销失败，而非用户侧的配置错误。它与以往侧重于 IAM 配置错误或权限提升的云安全研究形成对比；在此，平台本身未能响应用户的明确删除命令。它与其他分布式系统中关于“僵尸”凭证的研究（例如 AWS IAM 的最终一致性延迟）类似，但展示了更严重的持久性——密钥无限期保持活跃，而非仅在短暂的传播窗口内。

重要性

这是一个高严重性的供应链和基础设施安全问题。API 密钥是自动化云操作、CI/CD 流水线以及服务间身份验证的关键。如果删除不等于立即撤销，那么整个密钥轮换和事件响应策略将被破坏。攻击者若攻陷一个密钥并随后触发其删除（例如通过社会工程学或先前的入侵），则可以维持持久的未检测访问。这从根本上破坏了凭证生命周期管理的安全模型。

批判性视角

该论文的声明需要验证密钥持久性的确切范围和持续时间。目前尚不清楚这是否影响所有 GCP 服务还是仅特定服务，以及密钥是最终过期还是无限期保持活跃。研究人员尚未披露 Google 是否已承认或修补此问题。存在一种可能性，即该行为是一个未记录的功能（例如用于多区域部署中的延迟传播），而非纯粹的漏洞。实际可利用性取决于攻击者是否已经获取到密钥值。

后续关注

  • 展示实际影响的概念验证利用（例如使用已删除的密钥访问 GCP 服务）。
  • 关于云服务 SLA 是否应保证同步撤销的行业讨论。
  • 密钥管理工具（例如 HashiCorp Vault）的更新，以检测或缓解此类平台级延迟。
